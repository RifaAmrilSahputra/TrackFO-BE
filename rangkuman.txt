RANGKUMAN TEKNIS BACKEND TRACKFO
============================================================
Dokumentasi ini disusun berdasarkan analisis langsung terhadap source code yang tersedia di repository.

Catatan: Saat analisis, beberapa test integration/e2e sempat gagal akibat konfigurasi koneksi database (environment DATABASE_URL dan reachable host). Namun dokumentasi berikut berfokus pada desain arsitektur, perilaku endpoint, middleware, service, dan skema database berdasarkan source code.

============================================================
1) GAMBARAN UMUM SISTEM
------------------------------------------------------------
Nama Sistem:
- TrackFO Backend (trackfo-be) / “TrackFO Backend” (berdasarkan README.md)

Tujuan Sistem:
- Menyediakan REST API terpusat untuk mendukung aplikasi mobile TrackFO.
- Mengelola alur gangguan (issues), penugasan teknisi (assignments), tracking lokasi teknisi (tracking), dan pembuatan laporan penyelesaian pekerjaan (reports).

Permasalahan yang Diselesaikan:
- Manajemen gangguan dari tahap awal sampai selesai.
- Pengaturan siapa teknisi yang ditangani (manual/otomatis) berdasarkan status ketersediaan dan jarak.
- Pengumpulan lokasi teknisi secara periodik berbasis threshold perpindahan.
- Penutupan gangguan setelah laporan diverifikasi oleh leader teknisi.

Fungsi Utama Backend:
- Autentikasi & otorisasi berbasis JWT dan role-based access control (RBAC).
- CRUD gangguan (issues) untuk admin.
- Assignment teknisi ke gangguan (manual & otomatis) serta validasi transisi status.
- Tracking posisi teknisi.
- Pembuatan laporan penyelesaian oleh leader teknisi.
- Pengelolaan akun user dan profil teknisi.

Aktor / Role yang Terlibat:
- ADMIN
- TEKNISI

2) ARSITEKTUR SISTEM
------------------------------------------------------------
Arsitektur Backend Keseluruhan:
- Express.js sebagai HTTP server dan router.
- Struktur MVC-ish:
  - Routes (src/routes/*.route.js)
  - Controllers (src/controllers/*.controller.js)
  - Services / Business Logic (src/services/*.service.js)
  - Middleware (src/middlewares/*.js)
  - Utilitas (src/utils/*.js)
- Prisma ORM sebagai akses database (MySQL) melalui skema Prisma.

Hubungan Antar Komponen:
- Client (Mobile) → HTTP API (Express)
- Request masuk ke app.js → routes/index.js → route spesifik
- Middleware authGuard/authorizeRole melakukan validasi token + cek user/role
- Controller memanggil service
- Service melakukan transaksi dan query dengan Prisma
- Prisma melakukan query ke MySQL (container db pada docker-compose)

Pola Arsitektur:
- Layered architecture (routing-controller-service-data)
- Transaction boundary dengan prisma.$transaction untuk konsistensi bisnis
- RBAC + state validation (business rules status flow)

Diagram Alur (ASCII) - Request → Response:

[Client]
   |
   |  HTTP /api/<module>/<endpoint>
   v
[Express App (src/app.js)]
   |
   |  middleware: CORS, express.json
   v
[Router (src/routes/index.js)]
   |
   |  /auth, /users, /issues, /tracking, /reports
   v
[Route Handler]
   |
   |  middleware: authGuard (JWT + DB sync + isActive)
   |  middleware: authorizeRole (RBAC)
   v
[Controller]
   |
   |  panggil Service
   v
[Service]
   |
   |  Prisma query / transaction
   v
[MySQL Database]
   |
   |  hasil → format response
   v
[Controller → Express → Response]

Diagram Alur (ASCII) - Assignment & Status Flow:

Admin:
  POST /api/issues/:id/assign
      -> assignTeknisiToGangguan()
      -> create assignment(s)
      -> update gangguan.status='assigned'
      -> update teknisi.status='busy'

Teknisi:
  PATCH /api/issues/assignment/:assignmentId/status
      -> validateStatusTransition (assigned→on_the_way→working→done)
      -> update assignment.status
      -> jika done untuk semua assignment → gangguan.status='done'
      -> teknisi.status='available'

- bcrypt: hashing password untuk keamanan

7) AUTENTIKASI DAN AUTORISASI
------------------------------------------------------------
Login dan Token:
- Endpoint: POST /api/auth/login
- Implementasi: src/controllers/auth.controller.js → src/services/auth.service.js
- Proses:
  1) Validasi email & password tersedia
  2) services/auth.service.js:
     - find user by email + include roles
     - cek user.isActive
     - bcrypt.compare password
     - roles diambil dari user.roles (nama_role uppercase)
     - token dibuat menggunakan signToken() di src/utils/jwt.js

Pembuatan Token:
- src/utils/jwt.js signToken(payload)
  - payload minimal: { id, email }
  - jwt.sign(secret, expiresIn=JWT_EXPIRES_IN||'1h', issuer='laporketua-api')

Validasi Token:
- src/utils/jwt.js verifyToken(token)
  - jwt.verify(token, JWT_SECRET, {issuer})

Middleware authGuard:
- File: src/middlewares/auth.middleware.js
- Langkah:
  1) Cek header Authorization: Bearer <token>
  2) verifyToken(token)
  3) prisma.user.findUnique({ include roles & role:true })
  4) cek user exists
  5) cek user.isActive
  6) set req.user = {id, nama, email, roles:[...]}
  7) next()

Middleware authorizeRole:
- File: src/middlewares/role.middleware.js
- Langkah:
  - normalisasi role input dan role user
  - cek intersection
  - jika tidak cocok → 403

Role dan Akses:
- ADMIN:
  - dapat membuat/update gangguan
  - dapat assign teknisi manual/auto
  - dapat melihat semua gangguan, assignments, tracking, reports
  - dapat manajemen user/teknisi
  - dapat reset password teknisi
- TEKNISI:
  - dapat melihat daftar tasks gangguan (/issues/my/tasks)
  - dapat update status assignment mengikuti flow
  - dapat submit tracking lokasi
  - dapat membuat laporan (hanya leader assignment)
  - dapat update profil diri sendiri

Proteksi Endpoint:
- Pada semua route yang dilindungi, middleware authGuard dan authorizeRole dipasang sesuai kebutuhan.

9) BUSINESS PROCESS
------------------------------------------------------------
Proses Bisnis Utama:
A. Auth & Manajemen User
- Admin dapat membuat user/teknisi.
- User memiliki isActive flag untuk suspensi.

B. Lifecycle Gangguan
- Gangguan dibuat oleh ADMIN: status default 'open'.
- Setelah assign teknisi: gangguan.status → 'assigned'.
- Setelah leader submit laporan: semua assignment → done, gangguan.status → 'done'.

C. Lifecycle Assignment
- Status assignment mengikuti flow ketat:
  assigned → on_the_way → working → done
- Teknisi hanya dapat update status assignment miliknya.
- Jika seluruh assignment pada gangguan done → gangguan done + teknisi kembali available.

D. Tracking
- Teknisi mengirim latitude/longitude.
- Sistem menyimpan history hanya jika perpindahan >= 0.01 km (threshold).
- dataTeknisi status di-set menjadi busy saat tracking disubmit.

E. Laporan
- Hanya leader dari assignment aktif yang boleh submit.
- Saat laporan dibuat:
  - assignment semua teknisi pada gangguan diubah menjadi done
  - gangguan ditutup (done)
  - teknisi diubah status menjadi available

10) ALUR END-TO-END SISTEM
------------------------------------------------------------
1) Alur Login:
- Client → POST /api/auth/login {email,password}
- Backend:
  - authService.login()
  - verifikasi bcrypt password
  - signToken
  - return token + user/roles

2) Alur Autentikasi pada Request:
- Client menyertakan header Authorization: Bearer <token>
- authGuard memverifikasi JWT
- authGuard melakukan sync user + roles dari DB

3) Alur Pengelolaan Data Gangguan:
- Admin:
  - POST /api/issues
  - PATCH /api/issues/:id (tidak boleh jika status done)

4) Alur Transaksi Utama: Assignment (Manual/Auto)
- Admin:
  POST /api/issues/:id/assign
  - assignment.service.assignTeknisiToGangguan():
    - transaction
    - cek gangguan exists
    - pilih teknisi:
      - auto: bestTeknisi terdekat dengan haversine
      - manual: teknisiIds harus available
    - tentukan leader
    - buat assignment(s)
    - update teknisi status -> busy
    - update gangguan status -> assigned

5) Alur Tracking:
- Teknisi:
  POST /api/tracking {latitude,longitude}
  - tracking.service.recordTracking():
    - kalkulasi jarak dari posisi terakhir teknisi (MOVEMENT_THRESHOLD_KM = 0.01)
    - jika memenuhi threshold → create Tracking record
    - update dataTeknisi: latitude/longitude/lastSeen/status=busy

6) Alur Status Assignment:
- Teknisi:
  PATCH /api/issues/assignment/:assignmentId/status {status}
  - assignment.service.updateAssignmentStatus():
    - cek valid transition sesuai STATUS_FLOW
    - cek ownership assignment (teknisiId match)
    - jika status done:
      - cek semua assignment pada gangguan done
      - update gangguan done
      - update dataTeknisi menjadi available

7) Alur Pembuatan & Perubahan Data (Laporan):
- Leader:
  POST /api/reports {gangguanId,catatan,foto?}
  - laporan.service.createLaporan():
    - cek leader assignment aktif
    - create laporan
    - assignment.updateMany status=done
    - gangguan.update status=done
    - teknisi updateMany status=available

Diagram End-to-End (ASCII) - Penutupan Issue:

  POST /api/issues
    -> gangguan.status='open'

  POST /api/issues/:id/assign (manual/auto)
    -> assignments.status='assigned'
    -> teknisi.status='busy'
    -> gangguan.status='assigned'

  PATCH /api/issues/assignment/:assignmentId/status

  POST /api/reports

    -> semua assignment -> done
    -> gangguan -> done
    -> teknisi -> available

11) SERVICE DAN LOGIKA BISNIS
------------------------------------------------------------
Service 1: src/services/auth.service.js
- login(email,password)
  - find user + roles
  - cek isActive
  - bcrypt.compare
  - signToken
  - return {token,user:{id,nama,email,roles}}
- changeUserPassword(admin reset) / changeMyPassword(self)
  - find user
  - bcrypt compare oldPassword untuk changeMyPassword
  - bcrypt hash newPassword
  - userService.updatePassword()

Service 2: src/services/user.service.js
- createUserWithProfile(data)
  - transaction: create user + roles + optional teknisi profile
  - hash password
- updateFullUserProfile(id,data)
  - transaction:
    - update user nama/email
    - upsert DataTeknisi
- deleteUser(id)
  - soft delete (isActive=false)

Service 3: src/services/gangguan.service.js
- createGangguan(data, createdBy)
  - validasi judul/deskripsi
  - validasi lat/long
  - create Gangguan status default open
- updateGangguan(id,data)
  - cek exists
  - business rule: tidak boleh update jika status done
  - validasi status yang diizinkan
- getMyTasks(teknisiId)
  - cek dataTeknisi exists
  - ambil assignment by teknisiId include gangguan

Service 4: src/services/assignment.service.js
- assignTeknisiToGangguan(gangguanId, assignmentData, assignedBy)
  - transaction
  - auto: cari best teknisi terdekat (haversine) dari available
  - manual: validasi semua teknisiIds ada dan available
  - tentukan leader
  - buat assignment(s) status=assigned, update teknisi status busy
  - update gangguan status assigned

- getAssignmentsByGangguanId(gangguanId)
  - findMany assignment include teknisi user

- updateAssignmentStatus(assignmentId,status,teknisiId)
  - validasi status transition
  - cek ownership assignment
  - update assignment status dalam transaction
  - jika status done untuk semua assignment, update gangguan done dan teknisi available

Service 5: src/services/tracking.service.js
- recordTracking(userId, latitude, longitude)
  - validasi input
  - ambil DataTeknisi by userId
  - kalkulasi jarak threshold MOVEMENT_THRESHOLD_KM=0.01
  - transaction:
    - update dataTeknisi latitude/longitude/lastSeen/status=busy
    - jika shouldTrack → insert Tracking record

- getLatestTracking()
  - ambil semua DataTeknisi + include last tracking
  - map response {teknisiId,userId,nama,email,statusTeknisi,lastSeen,location,activeGangguan}

- getTrackingByTeknisi(teknisiId)
  - ambil tracking history take 100

Service 6: src/services/laporan.service.js
- createLaporan(teknisiUserId,data)
  - transaction
  - cek teknisi exists
  - cek gangguan exists dan tidak done
  - business rule: hanya leader assignment aktif yang boleh submit
  - create laporan
  - assignment.updateMany -> done
  - gangguan.update -> done
  - teknisi updateMany -> available

- getAllLaporan(), getLaporanById(), getMyLaporan()
  - fetch dengan include gangguan dan teknisi->user

12) VALIDASI DATA
------------------------------------------------------------
Lokasi Validasi:
- Controller kadang validasi minimal (mis. status wajib diisi)
- Service melakukan validasi business rule dan input numerik

Rules Validasi Utama:
- Auth
  - email/password wajib diisi
  - user harus isActive
  - password harus match bcrypt

- Gangguan
  - judul & deskripsi wajib trim
  - latitude/longitude wajib ada dan harus angka
  - status valid: open, assigned, on_progress, done
  - tidak boleh update gangguan jika status done

- Assignment
  - valid statuses assignment: assigned, on_the_way, working, done
  - status transition mengikuti flow ketat
  - hanya pemilik assignment boleh update status
  - manual assign: teknisiIds wajib ada dan harus tersedia (status available)
  - auto assign: teknisi available terdekat

- Tracking
  - latitude/longitude wajib dan harus angka
  - tracking disimpan hanya jika jarak >= 0.01 km

- Laporan
  - gangguanId & catatan wajib
  - harus leader dari assignment aktif

Dampak Validasi:
- Mencegah data inkonsisten (status flow tidak bisa skip/mundur)
- Mencegah submit laporan dari teknisi non-leader
- Mengurangi volume data tracking dengan threshold

23) RINGKASAN TEKNIS AKHIR
------------------------------------------------------------
Kesimpulan Cara Kerja Backend:
- Backend TrackFO adalah REST API Express dengan arsitektur berlapis (route-controller-service) dan Prisma ORM untuk MySQL.
- Autentikasi menggunakan JWT. Semua request terproteksi melewati authGuard untuk validasi token dan cek isActive user.
- Otorisasi menggunakan authorizeRole (ADMIN vs TEKNISI).
- Modul inti:
  1) Gangguan (issues): admin membuat dan meng-update status; teknisi mengambil tasks via assignment.
  2) Assignment: admin menetapkan teknisi (manual/auto) dan teknisi mengubah progress mengikuti flow status ketat.
  3) Tracking: teknisi mengirim lokasi; sistem menyimpan tracking hanya jika perpindahan melewati threshold.
  4) Laporan: leader assignment aktif submit laporan; sistem menutup gangguan dan membebaskan teknisi.

Alur data dari request sampai response:
- Request masuk → middleware authGuard & authorizeRole → controller → service → Prisma transaction/query → formatter response → balik ke client.

Komponen Inti:
- Express app: src/app.js, src/server.js
- Router modules: src/routes/*
- Middleware: src/middlewares/auth.middleware.js, src/middlewares/role.middleware.js, src/middlewares/error.middleware.js
- Business services: src/services/*
- JWT util: src/utils/jwt.js
- Skema database: prisma/schema.prisma
